# 第2回 演習プロンプト集（配布用ワークシート）

| 項目 | 内容 |
|------|------|
| 用途 | 第2回 演習「**Claude Code で仕様→実装＋セキュリティレビュー**」で**コピペして使うプロンプト・仕様テンプレート・記入用記録** |
| 進め方 | 詳しい手順は**スライド**を見ながら。**プロンプトはコピペ**して使用。**記録は本ファイルを Obsidian で開いて直接記入**（または自分の Vault にコピーして記入） |
| 環境 | 事前準備ガイドの通り **Obsidian Vault＋Claude Code を用意済み**（アカウントは社内手配済み）。**題材は極小**（CRUD 1つ／画面1つ／API 1つ）。**実データ・機密・個人情報は使わない**。**演習課題は各自セクターの題材を“自分の端末で最後までやり遂げる”**（講師環境では試せないため代行不可。Claude Code の一般的な使い方デモは別途あり得る）。うまくいかない場合は**個別サポートに相談** |

---

## 進め方（25分・ステップ0＋4ステップ）
⓪モデル設定＋サブエージェント整備（準備5分内）→ ①自セクターの極小題材を選ぶ → ②**受入基準付きの仕様**を Markdown で書く → ③Claude Code に**実装計画／ドラフト**を作らせる（spec＝source of truth）→ ④**セキュリティ観点**でレビュー。

## ステップ0：モデル設定＋サブエージェント整備（準備5分の中で実施）

1. **モデルと思考の深さを確認**：`claude` 起動後に `/model` → 既定の **Sonnet** のままでOK（難所のレビューだけ `/model opus` も可）。思考の深さ（effort）も**既定のまま**でOK。
   - なお、Claude 5 世代の最上位モデル **Fable** が、サブスクリプション各プランで **7/7（米国時間）まで**利用できます（週次上限の50%まで・`/model fable` で切替）。**研修当日が体験のチャンス**です（7/8 以降は一時的に使用量クレジット制へ移行、容量確保後にサブスクへ復帰予定 — Anthropic 公式発表 2026-07）。
2. **次のプロンプトをコピペして最初に実行**：

```text
OpusやSonnet、Haikuで処理が十分に可能な処理は、サブエージェントを立てて実行するようにしてください。このプロジェクト・セッションに限らず、他のプロジェクトでも同じようにサブエージェントを使うようにCLAUDE.mdなどを整備してください
```

3. Claude Code が CLAUDE.md に「軽い処理はサブエージェント（安価なモデル）へ委譲」というルールを書き込む提案をします。**内容を確認して承認**してください（＝「人が検証・承認」の実践）。

## ステップ2：受入基準付き仕様を自分で書く → Claude Code に磨かせる

**(1) まず自分で書く（5分）**：下のテンプレートを自分の Markdown ファイルにコピーし、「目的・入力・出力」を各1行＋**受入基準を2〜3個、自分の言葉で**書く（粗くてOK。ここが本演習の核です）。

```text
目的:
対象ユーザー:
入力:
出力:
制約:
受入基準:
- 
- 
セキュリティ上の注意:
未確定事項:
```

**(2) 次に Claude Code に磨かせる（3分）**：書いた下書きを Claude Code に貼り、続けて以下をコピペ。

```text
上記は私が書いた仕様の下書きです。
1. 曖昧な点・欠けている受入基準を、質問の形で指摘してください。
2. 私の回答を反映した清書版の Markdown 仕様を作成してください。
3. あなたが補った前提には行頭に【AI補完】と印を付けてください。
```

## ステップ3：実装計画またはドラフトを作らせる
```text
この Markdown 仕様をもとに、実装計画を作成してください。
ファイル構成、主要処理、テスト観点、未確定事項を分けてください。
実装に進む場合は、先に差分方針を説明してから最小ドラフトを作ってください。
```

## ステップ4：セキュリティ観点レビュー
```text
この仕様、実装計画、生成されたドラフトを、以下の観点でレビューしてください。
- IDOR

- 権限管理
- 機密情報
- ログ出力
- 外部API課金
リスク、理由、修正案を表にしてください。
```

## 発展：演習後に GitHub Spec-Kit で“型化”してコード改善（次の一歩／余力があれば当日）

Claude Code 単体の演習を終えたら、**同じ題材を GitHub Spec-Kit に載せ替え**て、構造化・改善まで体験します（**自分が作ったコードで** Spec-Kit の価値を実感する狙い）。

1. **Spec-Kit を導入**：演習フォルダで `specify init`（uv/uvx 経由・既存プロジェクト対応。詳細は公式 Quick Start）。→ Claude Code にスラッシュコマンドとテンプレが入る。
2. **演習のコード＋Markdown仕様を“Spec化”**：
   - `/speckit.specify`（何を・なぜ：目的／ユーザーストーリー／受入基準）
   - `/speckit.plan`（技術計画）→ `/speckit.tasks`（タスク分解）
   - 自作の Markdown 仕様を素材に、Spec-Kit の構造へ整える。
3. **見えた差分でコード改善**：構造化で浮いた不足（受入基準の漏れ・テスト未整備・非機能/セキュリティ要件）を `/speckit.implement` や追加プロンプトで改善。

> ねらい：**「自作 Markdown 仕様」と「Spec-Kit の構造化 spec」を比べる**ことで、型・テンプレ・移植性・拡張（コードレビュー／OWASP LLM 脅威モデリング／テスト追跡）の価値が具体的に分かる。Spec-Kit は **30+ ツール対応**（複数ツールでの仕様再利用・チームでの型統一に有効）。
